PHARM_RULES_SOLUTIONS.md
1. Определение формы
FORM = SOLUTION
Если лекарственная форма содержит жидкую дисперсионную среду и вещества полностью растворяются → применяется модуль SolutionsEngine.

2. Определение типа раствора
SolutionType =
1 TRUE_SOLUTION
2 CONCENTRATE_DILUTION
3 BURETTE_SYSTEM
4 NON_AQUEOUS_SOLUTION
Выбор ветки:
если ингредиенты вводятся как сухие вещества → TRUE_SOLUTION

если используются стандартные концентраты → CONCENTRATE_DILUTION

если используются бюреточные растворы аптеки → BURETTE_SYSTEM

если растворитель ≠ вода → NON_AQUEOUS_SOLUTION

3. Определение агрегатного состояния ингредиентов
Каждый ингредиент должен иметь:
presentationKind
Возможные значения:
solid
liquid
solution
concentrate
tincture
extract
oil
Классификация:
solid → входит в ΣSolids
liquid → входит в ΣLiquids
solution → входит в ΣLiquids
concentrate → входит в ΣConcentrates
tincture → входит в ΣLiquids
extract_liquid → ΣLiquids
extract_dry → ΣSolids

4. Основные формулы растворов
4.1 Расчёт массы сухого вещества
m = C × V / 100
где
m — масса вещества
C — концентрация %
V — конечный объём

4.2 Расчёт объёма концентрата
Если используется стандартный раствор:
V_conc = m / C_conc
или
V_conc = m × dilutionFactor
пример
1:10 → V = m × 10

5. Расчёт воды
Общая формула
V_water = V_total - ΣV_liquids
где
ΣV_liquids =
ΣV_concentrates +
ΣV_other_liquids

5.1 Правило ad
Если компонент имеет:
q.s.
ad
то его объём вычисляется:
V_component = V_total - ΣV_other_liquids

6. КУО (коэффициент увеличения объёма)
КУО учитывается только если:
вещество вводится как сухое
и
Σsolids ≥ 3%
Тогда:
V_displacement = Σ(m_i × KUO_i)
и формула воды:
V_water = V_total - ΣV_liquids - V_displacement

7. Бюреточная система
Если используется бюреточная система аптеки:
ingredient.presentationKind = concentrate
тогда:
KUO НЕ учитывается
расчёт выполняется:
V_water = V_total - ΣV_concentrates - ΣV_other_liquids

8. Стандартные аптечные концентраты
Примеры:
Glucose 50%
Caffeine-sodium benzoate 10%
Sodium bromide 20%
Для них используется правило:
V_conc = m / C

9. Классификация растворителей
Water
Ethanol
Glycerol
Oil
Propylene glycol
Dimexide
Если растворитель ≠ вода:
SolutionType = NON_AQUEOUS_SOLUTION

10. Правила смешивания
Порядок:
1 растворитель
2 концентраты
3 жидкие ингредиенты
4 доведение ad

11. Контроль доз
Если раствор предназначен для внутреннего применения:
DoseVolume = 15 ml
Количество доз:
N = V_total / DoseVolume
Доза вещества:
dose = total_mass / N

12. Контроль токсичности
Если вещество:
ListA
ListB
включается:
PoisonControlBlock
Проверяется:
разовая доза
суточная доза

13. Проверка прозрачности
Раствор должен быть:
прозрачный
без механических включений

14. Стабильность
Если вещество:
light_sensitive
используется:
dark_glass

15. Финальная проверка объёма
После смешивания:
V_final ≈ V_total
допустимое отклонение:
±2 %

16. Генерация ППК
PPC должен содержать:
исходные данные
нормализацию
математическое обоснование
расчёты
технологию
контроль
упаковку

Что важно
Эти правила позволяют движку:
	•	считать любой водный раствор
	•	считать бюреточные растворы
	•	правильно работать с ad
	•	учитывать КУО
	•	разделять твёрдые и жидкие компоненты

# CRITICAL_RULES_SOLUTIONS

## RULE_SOLUTION_017

### Несколько растворителей

Если в рецепте присутствуют несколько растворителей (например вода + спирт + глицерин):

```
V_total = Σ всех жидкостей
```

Вода **НЕ рассчитывается автоматически**, если она явно указана в рецепте.

Правило:

```
если Aqua purificata указана без ad
→ её объём фиксированный
```

---

## RULE_SOLUTION_018

### Приоритет ad

Если присутствует компонент:

```
Aqua purificata ad V_total
```

тогда:

```
V_water = V_total − ΣV_other_liquids
```

где

```
ΣV_other_liquids =
настойки
жидкие экстракты
концентраты
растворы
масла
```

---

## RULE_SOLUTION_019

### Твердые вещества в малой концентрации

Если

```
Σsolids < 3%
```

тогда

```
KUO не учитывается
```

и формула воды:

```
V_water = V_total − ΣV_liquids
```

---

## RULE_SOLUTION_020

### Растворы с сухими веществами >3%

Если

```
Σsolids ≥ 3%
```

включается **KUO-блок**.

```
V_displacement = Σ(m × KUO)
```

и вода:

```
V_water = V_total − ΣV_liquids − V_displacement
```

---

## RULE_SOLUTION_021

### Раствор из готового концентрата

Если ингредиент:

```
presentationKind = concentrate
```

то:

```
он не входит в Σsolids
```

и учитывается только как жидкость.

---

## RULE_SOLUTION_022

### Смешанные системы

Если рецепт содержит одновременно:

```
концентраты + сухие вещества
```

то порядок расчёта:

1. рассчитать объём концентратов
2. рассчитать KUO сухих веществ
3. вычислить воду

Формула:

```
V_water = V_total − ΣV_concentrates − ΣV_other_liquids − V_displacement
```

---

## RULE_SOLUTION_023

### Тинктуры и жидкие экстракты

Если ингредиент:

```
presentationKind = tincture
```

или

```
extract_liquid
```

он учитывается как:

```
ΣV_other_liquids
```

и уменьшает объём воды.

---

## RULE_SOLUTION_024

### Плотные растворители

Если растворитель:

```
glycerol
oil
propylene glycol
dimexide
```

используется правило **массового расчёта**.

```
m = ρ × V
```

где

```
ρ — плотность
```

---

## RULE_SOLUTION_025

### Растворы с несколькими концентрациями

Если рецепт содержит несколько концентраций:

пример

```
Natrii bromidi 2%
Magnesii sulfatis 3%
```

масса каждого вещества рассчитывается отдельно:

```
m1 = C1 × V / 100
m2 = C2 × V / 100
```

---

## RULE_SOLUTION_026

### Проверка физической реализуемости

После расчётов выполняется контроль:

```
ΣV_liquids ≤ V_total
```

если

```
ΣV_liquids > V_total
```

движок должен вернуть:

```
ERROR: impossible formulation
```

# Дополнительные проверки

Движок должен автоматически проверять:

```
1 Σsolids %
2 Σliquids
3 KUO необходимость
4 корректность ad
5 превышение объёма
6 дозы
```

# Почему эти правила важны

Без них ломаются рецепты:

* настойки + вода ad
* концентраты + сухие вещества
* несколько растворителей
* плотные растворители
* рецепты с двумя концентрациями

Это **самые частые ошибки студентов и фарм-калькуляторов**.

## RULE_SOLUTIONS_027 — Ввод по способу, а не только по форме

Каждый ингредиент раствора до расчёта обязан получить `introductionMode`.

Если `introductionMode` не определён:

* не применять скрытую технологию,
* выдать warning,
* считать только безопасную часть.

---

## RULE_SOLUTIONS_028 — Separate dissolution block

Если компонент требует отдельного предварительного растворения, движок должен строить 2 стадии:

1. предварительное растворение
2. введение в общую систему

И не смешивать это с прямым добавлением в общий объём.

Это важно для веществ, которые:

* медленно растворяются
* требуют особого растворителя
* требуют другого порядка внесения

---

## RULE_SOLUTIONS_029 — Add-at-end block

Если компонент относится к группе:

* настойки
* жидкие экстракты
* летучие вещества
* ароматические воды
* чувствительные к нагреву компоненты

то движок должен уметь помечать:
`addAtEnd = true`

И технология должна это отражать.

---

## RULE_SOLUTIONS_030 — Separate solvent requirement

Некоторые вещества растворяются не в основном растворителе, а сначала:

* в воде
* в спирте
* в глицерине
* в смеси

Для этого нужен блок:

```text
requiredPrimarySolvent
requiredPreDissolutionSolvent
```

Иначе движок будет пытаться растворять всё “в общей воде”.

---

## RULE_SOLUTIONS_031 — Order-sensitive ingredients

Есть вещества, для которых порядок внесения важен.

Нужен флаг:

```text
orderSensitive = true
```

и приоритеты:

* dissolve_first
* add_after_cooling
* add_last
* after_main_solution_ready

Без этого рецепт может быть математически верным, а технологически неверным.

---

## RULE_SOLUTIONS_032 — Heat policy

Нужен отдельный параметр:

```text
heatPolicy
```

варианты:

```text
allow_heating
mild_heating_only
no_heating
add_after_cooling
```

Потому что сейчас “не греть спирт” — это хорошо, но этого мало.
Нужно универсальное правило для всех компонентов.

---

## RULE_SOLUTIONS_033 — Volatility policy

Для летучих веществ нужен отдельный блок:

```text
volatilityPolicy
```

Например:

* minimize_open_exposure
* no_heating
* add_last
* avoid_filtration_if_possible

---

## RULE_SOLUTIONS_034 — Phase compatibility before math

Сначала проверка совместимости фаз, потом математика.

То есть если:

* вода + масло
* вода + жирный компонент
* слабосмешивающиеся жидкости

движок сначала должен решить:

```text
это вообще истинный раствор?
```

а уже потом считать воду.

Иначе он считает “раствор”, а по факту там эмульсия или расслоение.

---

## RULE_SOLUTIONS_035 — Solubility gate

Нужен флаг растворимости:

```text
solubilityClass
```

например:

* freely_soluble
* soluble
* sparingly_soluble
* practically_insoluble
* requires_co_solvent

Без этого движок может относить к true solution то, что вообще-то должно идти в суспензию или в особую ветку.

---

## RULE_SOLUTIONS_036 — Route-dependent technology

Один и тот же раствор для:

* внутреннего применения
* наружного
* глазных капель
* ушных
* носовых

может требовать разный контроль и упаковку.

То есть route должен влиять не только на label, но и на:

* технологию
* контроль
* упаковку
* предупреждения

---

## RULE_SOLUTIONS_037 — Final object class recheck

После всех расчётов движок должен ещё раз спросить себя:

```text
какой это итоговый объект на самом деле?
```

* true solution
* mixed liquid
* non-aqueous solution
* incompatible system
* unresolved technological case

Это вторая проверка после классификации. Очень полезная штука.

---

## RULE_SOLUTIONS_038 — Technology-capability warning

Если движок умеет считать математику, но не уверен в технологии, он должен писать не “готово”, а:

* math_resolved
* technology_partially_unresolved

Это очень важно для доверия.

---

## RULE_SOLUTIONS_039 — Ingredient behavior profile

Для каждого ингредиента в reference-базе хорошо бы иметь не только:
название / КУО / концентрацию,

но и:

```text
behaviorProfile:
- countsAsLiquid
- countsAsSolid
- affectsAd
- affectsKvo
- requiresSeparateDissolution
- addAtEnd
- heatPolicy
- volatilityPolicy
- lightSensitive
- filtrationPolicy
- phaseType
- compatibleWithWater
- compatibleWithAlcohol
- compatibleWithGlycerin
```

Вот это очень мощная штука.
Она и спасает от рецептов-“ломателей”.

# Что у тебя сейчас, скорее всего, не формализовано до конца

Вот самые опасные категории, которые часто ломают всё:

1. **настойки**
   они не просто liquid, они ещё уменьшают воду и часто идут ближе к концу

2. **жидкие экстракты**
   аналогично, но не всегда одинаково с настойками

3. **ароматические воды**
   их легко спутать с обычной водой или solvent/ad

4. **готовые растворы**
   часто ошибочно попадают в solids

5. **стандартные растворы**
   особенно если у них особая технология применения

6. **летучие компоненты**
   математика ок, технология ломается

7. **плотные неводные растворители**
   глицерин, масла, димексид — нужна отдельная ветка

8. **плохо растворимые вещества**
   они могут вообще выбрасывать рецепт из ветки solutions

9. **компоненты “вводить после охлаждения”**
   это очень частый реальный фарм-кейс

10. **компоненты, требующие отдельного растворения**
    без этого движок делает красивую, но фальшивую технологию

# Самая важная мысль

Тебе нужно описывать не только:

**что считать**

но и:

**как компонент ведёт себя в технологии**

Именно этого обычно не хватает, когда “какой-нибудь один рецепт всё ломает”.
